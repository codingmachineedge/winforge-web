Module Catalog · 模組目錄

WinForge Web ships 315 modules across 4 sections — every one with a real working implementation, no stubs — plus a browsable 895-tweak Windows catalog (22 categories). The catalog is data-driven from the original WinForge desktop suite's navigation tree and TweakCatalog, plus web-first additions.

Sections

SectionWhat lives there

CategoriesSystem tools that drive real Windows facilities through the built-in Rust backend — services, startup apps, connections, environment variables, drives, events, devices, battery, hosts, File Locksmith, File Browser, and many more

Windows 11The All Tweaks catalog — WinForge's 895 desktop tweaks across 22 categories, browsable and searchable in both languages

Filters: All / Web-capable / Native-only, fuzzy search with typo tolerance and match highlighting, grid or list view, favorites (pin ⭐) and a recents strip. The nav rail exposes Modules · Simulations · Reactor · Settings · About.

The Windows tweak catalog

The Windows 11 › All Tweaks section is a faithful, browsable port of WinForge's desktop TweakCatalog — 895 tweaks across 22 categories (Appearance, Explorer, Taskbar, Privacy, Performance, Network, Cleanup, Security, System, Debloat, Winaero, Vault, Network Pro, Windows 11 Advanced, Recipes, …). Each category opens a searchable, fully-bilingual list where every tweak shows a kind-appropriate control preview (toggle / action / slider / choice / colour / info) and its real metadata — kind, elevation (admin), destructiveness, and restart scope. A global search for any tweak name (e.g. "bitlocker") surfaces its category.

Registry writes are desktop-only, so — like the native modules — the tweaks render as a searchable reference in the browser and apply through the installed app. The data lives in src/data/tweaks.ts, generated straight from WinForge's own --export-docs exporter by tools/gen-tweaks.mjs.

Native modules & the backend

Native modules do their real work through the Tauri Rust backend — a command runner, a PowerShell runner (Windows PowerShell 5.1), typed filesystem commands, and a vetted-operation allowlist (run_op) for common Windows housekeeping. In a plain browser (npm run dev) the same UI renders with a note that live results need the installed app.

Highlight — File Browser (module.filebrowser): a full file manager with drive picker, breadcrumbs, size/date/attribute listing, new-folder / rename / copy / move, delete to the Recycle Bin (via in-process .NET — no separate helper), hidden-file toggle and a text preview pane, powered by dedicated Rust fs_* commands.

Trilingual UI

Every module is English / 粵語 (written Cantonese, Traditional Chinese) / bilingual, switched from the shell. A CI guard (src/i18n/moduleKeys.test.ts) scans every t('…') call in the source and fails the build if any key lacks either language — the catalog is enforced at **zero missing strings**.

Reactor Control Room · 反應堆主控室

The flagship of WinForge Web: a physics-based PWR nuclear power plant simulator presented as a full CRT-style control room console.

Core cutaway

The grain button opens a live vessel cross-section: 15 fuel-assembly columns coloured by local power (centre-peaked radial shape), control rods descending to the true insertion depth, coolant up-flow, and the axial power shape (a rod-skewed cosine). Readouts include peak fuel temperature, neutron flux and the peaking factor Fq.

Sound

Opt-in (off by default): an ambient reactor-hall hum that swells with power, neutron-detector clicks whose rate follows the source-range count rate, the alarm klaxon, and a SCRAM whoop. Everything is synthesized live with WebAudio — nothing is downloaded.

Quick start

  1. Press Warm Start — the plant comes up hot and critical with the turbine latched.
  2. Watch the turbine roll to 1800 rpm; the generator auto-syncs (SYNC) and megawatts

appear on the HUD.

  1. Try a SCRAM: rods drop, the strobe fires, the annunciator lights up — acknowledge with

ACK ALL, then investigate what tripped.

  1. Or start from cold iron: pumps on, feedwater up, mode Startup, dilute boron toward

criticality and withdraw rods slowly — the Reactor Protection System is watching.

What is simulated

The engine (src/reactor/) is a faithful TypeScript port of WinForge's C# ReactorSimService:

  • Six-group point kinetics (implicit Euler, 50 Hz sub-stepping) with a neutron source term
  • Reactivity feedback: rod S-curves (4 Westinghouse banks), soluble-boron differential worth,

Doppler, moderator temperature coefficient, xenon/samarium poisons, burnup defect

  • ANS-5.1 decay heat (23 groups + actinides), fuel-cycle burnup BOL→EOL
  • Thermal-hydraulics: fuel/coolant lumps, saturated pressurizer, RCP spin-up/coastdown with a

natural-circulation floor

  • Nuclear instrumentation (SR/IR/PR), P-6…P-10 permissives, Tech-Spec MODE 1–6, a full

Reactor Protection System trip set, and an inverse-kinetics reactimeter

  • Protection & ESF engines running behind the console and alarming into the annunciator:

rod-bank overlap program + Tavg/Tref auto control, PORV / code safeties / pressurizer relief tank (with the TMI-2 stuck-PORV drill), Appendix-G P/T limits + LTOP + PTS, safety injection + accumulators + MSSVs, containment + RCP seal-LOCA, the six Critical Safety Function trees

  • Turbine & balance-of-plant: governor roll, generator auto-sync, steam-dump valves,

SG level, breaker-gated electric output

  • Fuel factory (fabricate → load → burn → auto-discharge) with signed assemblies, and

power-generation credits

The full fidelity checklist against the C# source lives in docs/reactor-parity.md.

*Engineering numbers are tuned for plausibility and playability — this is a simulator game, not licensing-grade analysis.*

Development · 開發

Stack

  • Frontend: React 18 + TypeScript, Vite, react-i18next, PWA (offline-capable)
  • Desktop shell: Tauri v2 — Rust backend (src-tauri/) builds a Windows .exe / installer
  • Design: Material 3 shell (navigation rail + modal drawer, md-* token system, light/dark);

the reactor renders as a self-contained CRT console

  • Fonts/icons: bundled locally (@fontsource/*, Material Symbols) — no CDN calls, CSP-safe

Commands

npm install --legacy-peer-deps
npm run dev           # browser dev server (native panels degrade gracefully)
npm run tauri:dev     # desktop app, hot reload
npm run tauri:build   # Windows installer under src-tauri/target/release/bundle
npm run typecheck     # tsc --noEmit
npm test              # vitest

Layout

PathWhat
src/reactor/The physics engines: physics.ts (point kinetics core, framework-free), one pure module per protection/ESF subsystem, reactorAux.ts (the single coordinator that wires them onto the live sim), useReactorSim.ts (the only React bridge)
src/components/reactor/controlRoom/The Control Room console (HUD, mimic, gauges, annunciator, core cutaway, WebAudio)

docs/reactor-parity.mdReactor fidelity checklist vs the original C# engine

Engine architecture rules

  1. physics.ts stays framework-free; new reactor subsystems are separate pure

step(inputs, dt) → outputs modules wired only through reactorAux.ts.

  1. Modules never punt to the old desktop app — WinForge Web is the product.
  2. Everything user-facing is trilingual (EN / 粵語 / bilingual); the i18n guard fails CI on any

missing key.

  1. Native work goes through the Rust backend; in-process .NET (Add-Type via PowerShell) covers

Win32 APIs Rust std can't reach — no separate helper processes.

Testing

~760 vitest tests: per-subsystem physics suites (kinetics, CVCS, rods, relief, P/T, ESF, containment, CSF, turbine), coordinator integration scenarios (LTOP arming, stuck-PORV isolation, SI boration, seal-LOCA…), fuel factory + credits, i18n guards, registry guards.

Build from Source · 由原始碼建置

WinForge Web is a Tauri v2 desktop app with a React + TypeScript (Vite) frontend.

Prerequisites

  • Node.js 18+ and npm
  • Rust toolchain (rustup, MSVC host) + VS Build Tools / MSVC
  • WebView2 runtime (bundled on Windows 11)

Frontend (browser)

npm install --legacy-peer-deps
npm run dev        # Vite dev server at http://localhost:5199
npm run typecheck  # tsc --noEmit
npm test           # vitest
npm run build      # tsc --noEmit && vite build

In a plain browser isTauri() is false, so native panels render with a note that live results need the installed app; web-safe modules run fully.

Desktop app (Tauri v2)

npm run tauri:dev     # desktop app, hot-reload frontend + Rust backend
npm run tauri:build   # Windows .exe + NSIS/MSI installer under src-tauri/target/release/bundle

Regenerating the catalog & tweaks

node tools/gen-catalog.mjs "C:/path/to/WinForge"   # module catalog + the 22 tweak categories
node tools/gen-tweaks.mjs                           # the 895-tweak catalog from WinForge --export-docs
node tools/gen-registry-keys.mjs                    # registry-guard mirror

See also Development for the repo layout, engine rules and testing.

FAQ · 常見問題

Is this the same as the WinForge desktop app?

WinForge Web is its own product — a Tauri v2 Windows desktop app with a React + TypeScript frontend. It carries the WinForge catalog (315 modules + the 895-tweak Windows catalog) and the physics-based PWR reactor, and it also runs standalone in a plain browser for quick iteration.

Do the Windows tweaks and native modules actually work in a browser?

Native modules and the Windows tweaks perform real registry / service / filesystem operations through the built-in Rust backend, so they run in the installed desktop app. In a plain browser isTauri() is false, so those surfaces render as a **searchable, fully-bilingual reference** with a note that live results need the installed app. Web-safe modules (the Toolbox utilities, the reactor, etc.) run fully in the browser.

Where is the 895-tweak Windows catalog?

Under Windows 11 › All Tweaks — 22 categories browsable and searchable in English and 粵語, each tweak showing its real metadata (kind, elevation, destructiveness, restart scope). A global search for any tweak name (e.g. "bitlocker") surfaces its category. See Modules.

Where's the nuclear reactor?

The Reactor tab (or the ⚡ shortcut) opens the full CRT control-room console. See Reactor Control Room.

What is the Simulations tab?

Its own nav-rail tab collecting the reactor and its 18 reactor-powered industrial sims (Cake Factory, Grid Dispatch, Hydrogen, AI Cluster, HPC, smelter, data centre, collider, …) plus the Dashboard and Reactor Settings.

Why does npm install need --legacy-peer-deps?

The frontend pins a newer Vite than one dev plugin advertises support for; --legacy-peer-deps lets npm resolve the tree. It does not affect the build. See Build from Source.

Is it really bilingual?

Yes — every module title, keyword, label and tweak is **English / 粵語 (written Cantonese, Traditional Chinese) / bilingual**, switchable from the shell. A CI guard fails the build on any missing string in any language mode.
